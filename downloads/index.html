F7 should use a held-key guard so holding it does not spam dimension swaps and trigger lighting/render bugs.

Chat System

Chat is opened with T. Built-in commands are part of ChatOverlay. Mod commands should be read from loaded packs, not hardcoded.

/help — Show built-in commands and loaded modpack commands
/mods — Show modpack commands registered by loaded packs
/pos — Show position
/tp x y z — Teleport
/give id — Give block/item ID
/dim or /dimension — Switch dimension
/time 6000 or /time set noon — Set time
/summon baby | human | holographic human | boss — Built-in summon commands
/weather clear|rain|snow|thunder|random — Change weather
Rule: if a command belongs to a pack, it should appear only after that pack registers it.

What the Store Does

  • Downloads ui.store.
  • Parses the pack list.
  • Shows each pack as a card.
  • Downloads the selected zip into expansion_packs/.
  • Does not execute downloaded Java code.

Pack Anatomy

A modpack is a zip file with a manifest and optional data files.

Recommended structure

my_pack.zip
  pack_manifest.json
  content/
    commands.json
    blocks.json
    entities.json
    items.json
    recipes.json
    sheep_data.json
    sounds.json
    theme.json
    worldgen.json
  assets/
    music/
      magenta_stone.mp4
    sounds/
      click.wav
  README.txt

Required file

pack_manifest.json

The main entry point. It identifies the pack and points the loader to feature files.

Optional feature files

commands

Registers mod commands.

blocks

Registers block definitions.

entities

Registers entity definitions or data profiles.

items / recipes

Registers items and crafting recipes.

audio / theme

Registers sounds, music, or theme song metadata.

worldgen / biomes

Registers future terrain, biome, or spawn data.

Create Your Own Modpack

This is the full manual process for making a pack from scratch.

1. Make a folder

my_first_pack/
  pack_manifest.json
  content/
    commands.json
  README.txt

2. Write pack_manifest.json

{
  "packId": "my_first_pack",
  "name": "My First Pack",
  "version": "1.0.0",
  "author": "your_name",
  "description": "My first IRL Simulator expansion pack.",
  "entries": {
    "commands": "content/commands.json"
  }
}

3. Add commands.json

{
  "commands": [
    {
      "name": "/hello_pack",
      "usage": "/hello_pack",
      "description": "Tests that this modpack loaded.",
      "action": "message",
      "target": "Hello from my first pack!"
    }
  ]
}

4. Zip the contents

Select the contents inside my_first_pack/, then create:

my_first_pack.zip
Do not accidentally zip one extra parent folder if your loader expects pack_manifest.json at the zip root.

5. Test locally

expansion_packs/my_first_pack.zip

Then load/reload a world and type:

/mods
/help
/hello_pack

Create ui.store

ui.store is the online store list the game fetches.

{
  "title": "IRL Simulator Mod Store",
  "message": "Download expansion packs for IRL Simulator.",
  "packs": [
    {
      "name": "My First Pack",
      "description": "A test pack that registers /hello_pack.",
      "url": "https://yourname.github.io/modUI/packs/my_first_pack.zip"
    }
  ]
}
  • Each pack needs a name, description, and direct zip URL.
  • If the URL is wrong, the pack may appear but fail to download.
  • The store UI fetches the file path your Java code is configured to fetch.

Open both URLs in a browser first. If the browser cannot load/download them, the game cannot either.

Install / Use Mods In Game

Player flow

  1. Open IRL Simulator.
  2. Open Mod Store / Inbox.
  3. Wait for the loading spinner.
  4. Click Download on a pack.
  5. The zip downloads into expansion_packs/.
  6. Reload/load a world so the loader can read installed packs.
  7. Use /mods to check registered commands.
  8. Use /help to see built-in and modpack commands.

Manual install

1. Download pack zip manually.
2. Put it in expansion_packs/.
3. Start or reload the game.
4. Load a world.
5. Type /mods.

Integration Types

This page lists the kinds of things the game can register from packs if the loader supports them.

Blocks

IDs, names, colors, solidity, and light emission.

Entities

Entity profiles, display names, colors, base templates, and spawn data.

Items

Item IDs, names, icons, stack sizes, and usage behavior keys.

Recipes

Crafting grid patterns, inputs, outputs, and counts.

Commands

Command name, usage, description, safe action, and target.

Audio / Theme

Music files, sound effects, menu theme, boss theme, and ambience metadata.

World / Biomes

Terrain presets, biome names, spawn lists, colors, and generation options.

UI Themes

Colors, panel styles, accent colors, and menu metadata.

Block Integration

A block pack can describe blocks. The game can register them if the block loader supports the file format.

Example blocks.json

{
  "blocks": [
    {
      "id": 80,
      "name": "Ruby Block",
      "solid": true,
      "color": [180, 20, 45],
      "emission": 0
    },
    {
      "id": 81,
      "name": "Neon Blue",
      "solid": true,
      "color": [40, 140, 255],
      "emission": 12
    }
  ]
}

What the engine uses it for

  • id becomes the block ID for /give and placement.
  • color can be used by Chunk material color rendering.
  • emission can be used by Voxel light emission.
  • solid can be used by collision and block rules.
A block data file does not automatically work unless the engine has a block registry and the render/light/collision systems check it.

Entity Integration

An entity pack should register entity data, not random Java classes. The engine decides what templates it supports.

Example entities.json

{
  "entities": [
    {
      "type": "crystal_golem",
      "template": "blocky_humanoid",
      "displayName": "Crystal Golem",
      "bodyColor": [120, 220, 255],
      "headColor": [180, 245, 255],
      "health": 40,
      "speed": 0.018
    }
  ]
}

How it can work

  • type is the entity key.
  • template says which engine-side generic entity shape to use.
  • Colors customize the mesh.
  • Stats customize behavior if the engine supports them.
This keeps modpacks data-driven. The pack describes a golem; the engine provides the actual safe entity implementation.

Items + Recipes

Example items.json

{
  "items": [
    {
      "id": 120,
      "name": "Ruby Shard",
      "stackSize": 64,
      "color": [255, 60, 90]
    }
  ]
}

Example recipes.json

{
  "recipes": [
    {
      "type": "3x3",
      "pattern": ["RRR", "RRR", "RRR"],
      "keys": { "R": "Ruby Shard" },
      "output": "Ruby Block",
      "count": 1
    }
  ]
}

What the game needs

  • An item registry.
  • A recipe registry.
  • Crafting UI logic that checks registered recipes.
  • A mapping between item names/IDs and block outputs.

Sounds + Theme Packs

This is the missing theme-song/mod audio part.

Example theme.json

{
  "theme": {
    "menuMusic": "assets/music/magenta_stone.mp4",
    "bossMusic": "assets/music/boss_mom_theme.mp4",
    "clickSound": "assets/sounds/click.wav",
    "packAccentColor": [217, 67, 255]
  }
}

Example manifest entry

{
  "entries": {
    "theme": "content/theme.json"
  }
}

How it can be used

  • The menu can display the pack's theme song.
  • The Mod Store can preview music if the file exists.
  • The engine can choose menu music or boss music from loaded theme metadata.
  • Sounds can be referenced by name instead of hardcoding paths.
Large audio/video files can be heavy. Keep file sizes reasonable if hosting on GitHub Pages.

World + Biome Data

World/biome packs can describe future generation settings.

Example worldgen.json

{
  "worldgen": {
    "terrain": "hills",
    "surfaceBlock": "Grass",
    "undergroundBlock": "Stone",
    "treeChance": 0.08,
    "spawnAnimals": ["sheep"]
  }
}

Example biomes.json

{
  "biomes": [
    {
      "name": "Blue Forest",
      "grassColor": [80, 180, 255],
      "skyColor": [100, 190, 255],
      "animals": ["sheep"],
      "structures": ["small_house"]
    }
  ]
}

These files only work if the world generator is coded to read registry data.

Animal Data

Animal data is a profile/database entry. It does not spawn anything by itself.

Example

{
  "animals": [
    {
      "type": "sheep",
      "names": ["Wooly", "BaaBaa", "Cloud", "Fluffy"],
      "colors": [[235,235,235], [255,230,240], [210,240,255]]
    }
  ]
}

Meaning

  • type is the animal key/name.
  • names is a random name pool.
  • colors is a list of RGB colors.
  • A command or world system must reference the animal type before the data matters.

Troubleshooting Mods

/mods says no commands

  • Check that the zip is in expansion_packs/.
  • Check that the pack has command data.
  • Check that the loader runs when a world loads.
  • Check that your command file path matches the manifest or loader expectation.

Download fails

  • Open the pack URL in a browser.
  • Make sure the zip exists online.
  • Make sure ui.store has the correct URL.

Pack loads but does nothing

  • The pack may only contain data.
  • The engine may not support that data type yet.
  • The command action may not be implemented.

Wrong zip structure

  • pack_manifest.json should be at the zip root if your loader expects root manifest.
  • Do not accidentally zip one extra parent folder.
